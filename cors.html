Web Security / HTTP

CORS —
кога блокира?

Пълен справочник: кои браузърни механизми минават през CORS проверка и кои не.

01 —

type="module"

ES модулите винаги минават през CORS, дори и да са обикновен <script> таг.

HTML
<!-- Обикновен script — без CORS --> <script src="https://cdn.com/lib.js"></script> <!-- Module — минава през CORS! --> <script type="module" src="https://друг-домейн.com/mod.js"></script>
❌ CORS грешка ако сървърът няма Access-Control-Allow-Origin
Защо? Модулите са нов стандарт и умишлено са проектирани с CORS — по-сигурно поведение от legacy <script>.
02 —

fetch() и XMLHttpRequest

Всяка програмна HTTP заявка към друг домейн минава задължително през CORS.

JavaScript
// ❌ CORS грешка — сървърът трябва да позволи fetch('https://друг-домейн.com/api/data.json') // Браузърът изпраща preflight OPTIONS заявка: OPTIONS /api/data.json HTTP/1.1 Origin: https://моя-сайт.com Access-Control-Request-Method: GET // Сървърът трябва да отговори с: Access-Control-Allow-Origin: * Access-Control-Allow-Methods: GET, POST
❌ Блокирано без правилните хедъри
03 —

Атрибутът crossorigin — капан

Добавянето на crossorigin към всеки таг нарочно активира CORS режим, дори там където по подразбиране няма CORS.

HTML
<!-- Без crossorigin — работи нормално --> <script src="https://cdn.com/lib.js"></script> ✓ OK <!-- С crossorigin — CORS се активира! --> <script crossorigin src="https://cdn.com/lib.js"></script> ❌ <!-- Същото за img --> <img crossorigin src="https://cdn.com/photo.jpg" /> ❌
Защо изобщо се ползва? За да виждаш пълни stack traces в конзолата при JS грешки от CDN. Без него браузърът крие детайлите на грешката за сигурност. Но тогава CDN-ът трябва да поддържа CORS.
04 —

CSS @font-face от друг домейн

Шрифтовете са специален случай — винаги минават през CORS, за разлика от изображения и скриптове.

CSS
/* ❌ Не работи ако сървърът няма CORS хедър */ @font-face { font-family: 'MyFont'; src: url('https://друг-домейн.com/font.woff2'); } /* ✓ Google Fonts работи защото те имат: */ Access-Control-Allow-Origin: *
❌ Шрифтът не се зарежда без CORS
05 —

Canvas + изображения от друг домейн

Зареждането на изображението работи — но щом се опиташ да четеш пиксели, canvas се „замърсява" (tainted) и блокира.

JavaScript
const img = new Image(); img.src = 'https://друг-домейн.com/photo.jpg'; ctx.drawImage(img, 0, 0); ✓ Показва се // Но след това: canvas.toDataURL(); ❌ SecurityError: Tainted canvas ctx.getImageData(...); ❌ SecurityError: Tainted canvas // Решение: добави crossorigin на img И CORS на сървъра img.crossOrigin = 'anonymous'; ✓ Работи с CORS хедър
⚠️ Зарежда се, но четенето на пиксели е блокирано

Механизъм CORS? Бележка
<script src> ❌ Не Legacy — предхожда CORS
<script type="module"> ✅ Да Нов стандарт, умишлено с CORS
<script crossorigin> ✅ Да Атрибутът активира CORS режим
fetch() / XMLHttpRequest ✅ Да Всяка програмна заявка
<img src> ❌ Не Зарежда се свободно
<img> → Canvas четене ✅ Да getImageData / toDataURL блокирано
<link rel="stylesheet"> ❌ Не CSS файлове са свободни
@font-face в CSS ✅ Да Шрифтовете винаги минават CORS
<video> / <audio> ❌ Не Зареждат се свободно

Златното правило

  • ✓
    Зареждане на ресурс (script, img, video, css) — без CORS, освен шрифтове и модули.
  • ✗
    Четене на съдържанието програмно (fetch, XHR, getImageData) — винаги CORS.
  • ✗
    crossorigin атрибут — нарочно активира CORS дори там където няма.
